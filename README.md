## 实验结果评估
1.所有实验的协同设计都取得了最好的成绩，这说明结构信息和序列信息能够有效互补。

2.实验 2 的协同设计相对于实验1的 0.2918 退化至 0.4752，且 Struct-Only 彻底崩盘（2.0040）

H_struct 做 Query 更好 
在抗体设计中，用结构特征 (H_struct) 去查询并提取相关的序列进化特征 (H_seq) 是符合物理与生物逻辑的。结构作为锚点，能够精准地从庞大的序列潜空间中捞出有用的生化信息。

H_seq 做 Query 会失败
如果用序列去查询结构，模型相当于在以 1D 序列为核心来重构特征。模型似乎陷入了偷懒模式——它极度依赖序列特征，导致结构编码器（EGNN）在联合训练中被边缘化，梯度更新不充分，进而导致 Struct-Only 模式下性能断崖式下跌（2.0040）。协同设计的表现 (0.4752) 也几乎等同于单序列表现 (0.4760)，说明此时模型完全没有利用到结构信息。

3.结构特征作为查询信息可以起到主导作用，使协同设计达到最好的效果。

实验1结构作为查询时仅结构可以达到0.3605，协同设计可以达到0.2918，仅序列可以达到0.8613

实验2序列作为查询时仅序列可以达到0.4760，协同设计可以达到0.4752，仅结构 2.0040，崩盘了

4.预训练序列模型的重要性
设置差异： 序列编码器从 预训练的 ESM-2 (35M) 换成了 随机初始化的 Transformer (12M)。
结果对比： 

Seq-Only: 0.8613 (预训练) 变差至 2.1022 (无预训练)

Struct-Only: 0.3605 (预训练) 与 0.3516 (无预训练) 几乎持平

协同设计: 0.2918 (预训练) 变差至 0.3477 (无预训练)

小数据场景下的序列模型严重依赖预训练： 蛋白质序列空间极其庞大。在仅有约 10,000 条抗体数据（diffab_sabdab）的微调集上，从零开始训练的 Transformer (12M) 根本无法学习到有效的蛋白质语言规律，导致 Seq-Only 性能极差 (2.1022)。

EGNN 的数据高效性： 即使去掉了强大的 ESM-2，实验 3 中的 Struct-Only 依然保持了很好的性能 (0.3516)。这证明基于图神经网络（EGNN）的三维结构编码器具有极强的归纳偏置和旋转平移等变性，只需极少的数据（1万条）就能从零学会提取空间结构特征。

协同设计的上限被序列短板拉低： 在实验 3 中，由于序列编码器太弱，它不仅没有帮到结构编码器，反而拖了后腿。协同设计的成绩 (0.3477) 仅仅是勉强微弱优于纯结构设计 (0.3516)，完全失去了实验 1 中大幅提升的协同效果。
